Ruby Classes: How to Pick a Restaurant

September 27th, 2014

Ruby is a powerful programming language which, due to its focus on productivity and good user interface, is also relatively accessible to beginners. Ruby is an object-oriented programming language: almost everything in Ruby - Integers, Strings, Arrays, Hashes - is an object that can understand specific messages. A message sent to an object corresponds to a method, a routine that the object can initiate, allowing objects to store data, interact with each other, or perform any number of complicated actions.

But how do objects 'know' what they can or can't do? Where are the rules? What makes one different from the other? To answer these questions, you have to understand Classes.

A Class is a group of behaviors or attributes defined by a programmer. Every object is a part of a Class, referred to as being an 'instance' of a Class, and objects can only be instances of one Class. Ruby comes with a number of built-in Classes - such as 'String' - but what makes Classes so useful is the ability to define your own. Classes can, in turn, have sub-Classes that have their own properties, in addition to inheriting the properties of the parent Class.

This all probably seem complicated, but it's in many ways similar to how we think of things in our daily life. Imagine you're at a friend's place and you decide to go to dinner; you don't know what you want, so you start throwing out names:

class Restaurant
  def initialize(name, distance, cuisine)
    @name = name
    @distance = distance
    @cuisine = cuisine
  end

  def decisions
    puts "How about #{@name}?"
    puts "Its about #{@distance} away."
    puts "Are you in the mood for #{@cuisine}?"
  end

  def style
    dinner
  end
end

You decide you want to go to a place that will serve food (Restaurant) and base your decision on a variety of factors common to all Restaurants: How far away is it? What food does it serve? Have I heard of it? In Ruby, this information is set in the 'initialize method' as instance variables, each of which is available to the rest of the methods in the class and differs for each instance of that class. Just as you might go over each factor for each restaurant, we can do the same in Ruby by creating instances of this Class for each restaurant:

mcdonalds = Restaurant.new("McDonald's", "1 Mile", "Burgers")
shakeshack = Restaurant.new("Shake Shack", "2 Miles", "Burgers")
umami = Restaurant.new("Umami Burger", "1 Mile", "Burgers")


This Class allows us to group together a number of Restaurants with similar attributes: all Restaurants have a name, are a certain distance away, and serve a specific type of food. Still Restaurant is a broad category; these 4 instances of Restaurant have a few things in common, but they can also be divided further:

class Fastfood < Restaurant
  def style
    puts "It's cheap."
    puts "We can bring it back here."
    puts "We definitely have time to go there."
  end
end

class Fastcasual < Restaurant
  def style
    puts "Its not too expensive."
    puts "We can bring it back here or eat there if you want."
    puts "They're pretty fast, but the line might be long."
  end
end

class Sitdown < Restaurant
  def style
    puts "It's a bit expensive, but you can find something cheap."
    puts "Do you want to sit down somewhere?"
    puts "How much time do you have? There might be a wait."
  end
end


In this way, we can further differetiate objects that still share common attributes with all other objects. McDonald's and Umami Burger aren't much alike, but you can get a hamburger at both. If we want to figure out if we want to go to, say, Chipotle instead:

chipotle = Fastcasual.new("Chipotle", "half a mile", "Mexican")
